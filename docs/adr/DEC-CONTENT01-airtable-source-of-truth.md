# Airtable as the content source of truth, fetched at build time

Initial planning assumed a local CSV file + a `public/audio` folder as the content authoring workflow, edited by hand and committed to git. Ben flagged that updating content (especially adding new lessons) needs to be as convenient as possible, which a hand-edited CSV plus manual mp3 file placement doesn't deliver — no mobile editing, no drag-and-drop audio, and every change requires touching the filesystem directly.

**Decision**: Sentences (Amis text, Zh translation, Lesson, Section, Order, Pair Tag, and the audio file as a native Attachment) live in an Airtable base (`Nanamen`, base id `app8MvGUBu4Xg9HOR`, table `Sentences`) instead of a repo-local CSV. A `scripts/sync-content.mjs` script, run as an npm `prebuild`/`predev` hook (so it always runs before `next build`/`next dev` — no manual step, content updates take effect on the next deploy automatically), fetches every record via the Airtable REST API, **downloads each audio attachment into `public/audio/...`**, and writes `content/generated/sentences.json`. The sync preserves the attachment's audio extension when possible (for example `.webm` from SashaWaves/Vercel Blob, or `.mp3` from Airtable/E樂園 imports). The Next.js app only ever reads the generated JSON and local `/audio/...` paths — it has no runtime dependency on Airtable.

Audio is downloaded rather than linked directly: confirmed by testing that Airtable serves attachments via `v5.airtableusercontent.com` signed, time-limited URLs, not stable links. Embedding those directly would mean audio silently breaking between deploys whenever a URL's signature expires before the next rebuild.

**Consequences**: The *build* now depends on Airtable's API availability — a build fails if Airtable is unreachable or the API key is missing/invalid, but a already-deployed app keeps working regardless of Airtable's status (no runtime calls). Requires an Airtable Personal Access Token stored as `AIRTABLE_API_KEY` in both `.env.local` (local dev) and Vercel's environment variables (production/preview). `content/generated/` and the downloaded files under `public/audio/` are build artifacts (gitignored, regenerated every build), not committed. This is the first external service dependency in an otherwise dependency-free app; local runtime persistence (weak items, suspended pairs — see [DEC-AUTO01](DEC-AUTO01-weak-items-localstorage.md)) remains client-side localStorage, unaffected by this decision.
