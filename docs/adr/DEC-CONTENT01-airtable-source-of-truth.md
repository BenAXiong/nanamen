# Airtable as the content source of truth, fetched at build time

Initial planning assumed a local CSV file + a `public/audio` folder as the content authoring workflow, edited by hand and committed to git. Ben flagged that updating content (especially adding new lessons) needs to be as convenient as possible, which a hand-edited CSV plus manual mp3 file placement doesn't deliver — no mobile editing, no drag-and-drop audio, and every change requires touching the filesystem directly.

**Decision**: Sentences (Amis text, Zh translation, Lesson, Section, Order, Pair Tag, and the audio file as a native Attachment) live in an Airtable base (`Nanamen`, base id `app8MvGUBu4Xg9HOR`, table `Sentences`) instead of a repo-local CSV. The Next.js app fetches this base directly via the Airtable REST API in Server Components at build time (Next's fetch caching means it's called once per build, not per request), so there's no separate `content:build` script or generated JSON file — content updates take effect on the next deploy automatically. Audio attachments are downloaded and referenced from Airtable's hosted URLs (or re-hosted under `public/audio` if attachment URL stability becomes an issue later — not needed for MVP).

**Consequences**: The app now depends on Airtable's API availability at build time — a build fails if Airtable is unreachable or the API key is missing/invalid. Requires an Airtable Personal Access Token stored as `AIRTABLE_API_KEY` in both `.env.local` (local dev) and Vercel's environment variables (production/preview). This is the first external service dependency in an otherwise dependency-free app; local runtime persistence (weak items, suspended pairs — see [DEC-AUTO01](DEC-AUTO01-weak-items-localstorage.md)) remains client-side localStorage, unaffected by this decision.
